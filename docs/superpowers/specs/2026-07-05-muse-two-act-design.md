# muse 两幕剧 · 设计规格（盲区扫描 + 圆桌深挖 + 对抗审查）

日期：2026-07-05　状态：待用户终审
上游讨论：本文件固化 2026-07-05 会话中逐节确认的设计决定。

## 1. 背景与目标

用户核心场景：**启动 App → 输入论文主题/困惑 → 快速拿到想法与反对观点，最重要的是非显而易见的切入点、跨学科理论与方法——发掘"未知的未知"**。论文最重要的是视角与方法（主线），其余是组织问题；发现主线后，材料组织交给既有 xw-writing 流水线。

三个并列目标：
1. **发现力**：每次运行大概率产出 1-2 个真正意外的角度（机制见 §4-§6）；
2. **好看好用**：现代感视觉，秒级可用的首屏；
3. **联动**：产物可被 paper-annotator、xw-writing 技能链、其他 agent 直接消费。

设计参照 Thariq《Finding Your Unknowns》框架：盲区扫描（Blind Spot Pass）、"N 个截然不同方向让用户反应"、一次一问访谈、交付前对抗核查。

**期望管理**：本工具是"提高意外发现命中率的机器"，不是命中保证器。四层发现机制叠加，但不承诺每次必有惊喜。

## 2. 产品形态总览

- **PaperMuse.app**（macOS 14+）：SwiftUI 原生壳（窗口、菜单栏、进程管理、退出清理——沿用 v0.1 已建部分）+ **WKWebView 内容画布**渲染本地 web UI（视觉上限与迭代速度优先；无外网依赖，全部本地资源）。
- **muse_server.py**（FastAPI :8765，沿用扩展）：新增盲区扫描与对抗幕接口；继续单会话 + 锁串行模型。
- 顶部模式切换：`[构思幕 | 对抗幕]`。
- 所有产物落 `$PAPER_MUSE_OUTPUT_DIR/docs/agents/muse/`（anamra 注入论文目录时即落进该论文）；env 未设时兜底 `<repo>/results/muse/<主题>/docs/agents/muse/`，与 v0.1 行为连续。

## 3. 信息架构与主流程

```
启动 App
  └─ 构思幕（默认）
       ├─ 输入主题 → 【盲区扫描】30-60 秒 → 切入点卡片墙（首屏产品）
       │     └─ 点卡片 →（可选）圆桌深挖：以该切入点为种子起 Co-STORM
       └─ 产物：perspectives.md / questions.md / mindmap.md / sources.md
  └─ 对抗幕
       ├─ 选草稿（扫描论文目录 *.md，含 01_成品稿/）→ 红队审查
       └─ 产物：failure-points.md
```

## 4. 盲区扫描层（新建，首屏核心）

**输入**：主题或研究困惑（一句话到一段话）。
**过程**（30-60 秒预算）：
1. **多模型合议**：同一"视角枚举"提示词并行发给 DeepSeek + OpenAI + Gemini（可用几家发几家）；
2. 每家枚举跨学科理论/方法/框架候选（法经济学、组织社会学、STS、复杂系统、公共选择……不限于法学）；
3. 合并去重后，对每个候选做一次检索验证（复用多源检索层），拿到文献锚点；
4. **离群标亮**：仅一家模型提出的角度单独标记——离群处是 unknown unknowns 的藏身处。

**输出**：卡片墙，每张卡字段：
- 理论/方法名；一句话机制；**为什么非显而易见**（相对该主题的常规文献）；文献锚点（真实检索结果 URL/条目）；意外度（离群/共识）；来源模型。
- 卡片间**强制相异**（学科、方法论、规范/实证、时间尺度等维度错开），拒绝同一角度的变体。

**落盘**：`perspectives.md`（全部卡片）+ `sources.md`（锚点）。

## 5. 构思幕 · 圆桌深挖层（沿用 Co-STORM，扩展）

- 从卡片进入：以选中切入点为种子话题起 Co-STORM（热身 1-3 分钟可接受，因为首屏已有产出）。
- **钉死席位**（与自动发现的领域专家并列）：
  1. **第一性原理专家**——persona 取用户"第一性原理"三条提示词（回到根本、拆到最小可验证单元、能答"为什么"而非只答"怎么做"）；
  2. **跨学科猎人**——只负责把外域理论/方法拖上桌并论证适配性。
- 检索接地：多源检索层（Tavily / Perplexity / Mixed / Jina 全文增强），后续可加 Zotero VectorRM。
- 主持人机制保留并强化（从"检索到但未使用的信息"造问题＝信息层盲区挖掘）。
- 产物追加：`questions.md`（主持人问题清单，带文献依据）、`mindmap.md`。

## 6. 对抗幕（新建，轻量引擎，不套 Co-STORM）

- **草稿来源**：扫描 `$PAPER_MUSE_OUTPUT_DIR` 下 .md（含 `01_成品稿/`），App 内列表选择。
- **流程**：抽取中心主张（逐条）→ 每条生成 3-5 个最可能崩的失败点 → 逐条用检索层证伪/佐证 → **无验证证据的标"未决"，不许放行**。
- persona 纲领取用户"对抗式审查"三条提示词（以苛刻审稿人姿态攻击自己的成果、列 3-5 失败点、拒绝无证据的"看起来没问题"）。
- **实现**：独立轻量循环（读稿→拆主张→证伪），复用检索层与 LM 配置；不用 Co-STORM（其热身/思维导图/话题导向对"审一份稿"是错配）。
- **落盘**：`failure-points.md`（主张、失败点、证据链接或"未决"标记、建议动作）。

> 用户提示词以其原文为准；本 spec 中为要旨转述，实施时预留 persona 文本槽位供用户粘贴原文。

## 7. 多服务商

- 横切参数 `provider ∈ {deepseek, openai, gemini}`：一键整体切换六角色模型（仓库已有 OpenAIModel/Gemini 适配，2026 模型名以 README 顶部为准）。
- 可选**混编**：高频闲聊角色用便宜快模型，主持人提问/知识库综合用强模型。
- 多模型合议（§4）独立于此开关——扫描层始终尽可能多家并行，这是发现机制而非成本选项。

## 8. 文件契约（联动的唯一接口）

`docs/agents/muse/`：`perspectives.md`、`questions.md`、`mindmap.md`、`failure-points.md`、`sources.md`。
消费者：`grill-with-docs`（吃 questions 逐条拷问）、`to-prove`/`diagnose`（吃 failure-points）、paper-annotator（把失败点锚回稿面）、任何后续 agent。联动靠文件约定，不靠共享界面。

## 9. UI / 审美路线

- SwiftUI 壳保留：窗口、生命周期、muse_server 拉起/复用/退出清理（v0.1 代码沿用）。
- 内容画布 = WKWebView 加载本地 web UI（muse_server 静态托管或本地文件）；卡片墙、圆桌流、对抗报告全部 web 渲染。
- **设计流程**：按"N 个截然不同设计方向 → 用户反应 → 收敛"执行；先出 4 个方向的静态 HTML mock 供用户挑选，再实现选中方向。本地预览-截图-迭代工具链走通视觉回路。
- 视觉基调（待 mock 阶段确认）：现代、编辑器级排印、明暗双主题。

## 11. 非目标

- 不生成论文正稿文字；muse 产出的一切文本只作构思素材（文字"活人感"归 humanize 等技能）。
- 不做云端/多用户/移动端。
- 不在本期做 Zotero VectorRM 混检与思维导图可视化（列为后续单独计划）。

## 12. 验收标准

1. 冷启动到卡片墙 ≤ 90 秒（含多模型合议与检索验证），每次 ≥ 6 张卡且含 ≥ 1 张离群标亮；
2. 卡片→圆桌深挖→`questions.md`/`perspectives.md` 落盘齐全，grill-with-docs 能直接消费；
3. 对抗幕对一份真实草稿产出 `failure-points.md`，每个失败点带证据链接或"未决"标记，无凭空断言；
4. `provider` 三选一均可完整跑通构思幕与对抗幕；
5. App 视觉经用户按"4 方向反应法"定稿后实现，用户认可"好看、现代"；
6. 全程无终端操作：双击/点按启动，产物在访达一键可达。
