# muse 两幕剧 · 设计规格 v2（盲区扫描 + 圆桌深挖 + 对抗审查）

日期：2026-07-05　状态：**v2 审定**（用户委托对抗式审阅，8 项修订已并入；v1 见 git 历史 306473b）
persona 原文槽位：待用户提供「第一性原理」「对抗式审查」两组提示词原文后替换要旨转述版。

## 1. 背景与目标

用户核心场景：**启动 App → 输入论文主题/困惑 → 快速拿到想法与反对观点，最重要的是非显而易见的切入点、跨学科理论与方法——发掘"未知的未知"**。论文最重要的是视角与方法（主线），其余是组织问题；发现主线后，材料组织交给既有 xw-writing 流水线。

三个并列目标：
1. **发现力**：不止"角度存在"，还要"角度在用户的学术场（中文法学）里未被做烂"——新颖性是相对于领域文献与用户认知框架的，不是相对于模型语料的；
2. **好看好用**：现代感视觉，首批结果 20 秒内可见；
3. **联动**：产物可被 paper-annotator、xw-writing 技能链、其他 agent 直接消费。

设计参照 Thariq《Finding Your Unknowns》框架：盲区扫描、"N 个截然不同方向让用户反应"、一次一问访谈、交付前对抗核查、**给模型你的起点**。

**期望管理**：本工具是"提高意外发现命中率的机器"，不是命中保证器。

## 2. 产品形态总览

- **PaperMuse.app**（macOS 14+）：SwiftUI 原生壳（窗口、菜单栏、进程管理、退出清理——沿用 v0.1）+ **WKWebView 内容画布**渲染本地 web UI（无外网依赖）。
- **muse_server.py**（FastAPI :8765，沿用扩展）：新增盲区扫描（增量出卡）与对抗幕接口。
- 顶部模式切换：`[构思幕 | 对抗幕]`。
- 所有产物落 `$PAPER_MUSE_OUTPUT_DIR/docs/agents/muse/`（anamra 注入论文目录时即落进该论文）；env 未设时兜底 `<repo>/results/muse/<主题>/docs/agents/muse/`。
- 主题输入框默认带出 anamra 当前论文目录可推断的题目（有 PAPER_MUSE_OUTPUT_DIR 时读该目录近期 md 标题作预填建议）。

## 3. 信息架构与主流程

```
启动 App
  └─ 构思幕（默认）
       ├─ 输入主题（+可选研究者画像）
       │    → 【盲区扫描】流式出卡：首批 ≤20s，全墙 ≤90s
       │    → 切入点卡片墙（三类卡 × 新颖性分类 × 密度差信号 × 三键反馈）
       │    └─ 点卡片 → 圆桌深挖（Co-STORM 以该卡为种子）或 送对抗幕无稿模式
       └─ 产物：perspectives.md / questions.md / sources.md / profile.md / angle-feedback.json
  └─ 对抗幕
       ├─ 有稿模式：选草稿（扫论文目录 *.md）→ 红队审查 → failure-points.md
       └─ 无稿模式：攻击一个主线句（来自卡片或手输）→ 同格式输出
```

## 4. 盲区扫描层（新建，首屏核心）

**输入**：
- 主题或研究困惑（一句话到一段话）；
- **研究者画像**（可选，5 行以内：领域/已有立场/熟悉的理论/本篇的困惑）——首次填写后落 `docs/agents/muse/profile.md`，该论文后续自动复用；离群与新颖性判断以画像为参照系（发掘的是**相对于你**的未知）；
- **已知角度抑制表**：被用户标过「已知」的角度（`angle-feedback.json`）在本论文后续扫描中不再出现。

**流程**（流式，首批卡 ≤20 秒）：
0. **第一性原理拆解**：先把主题拆成 3-5 个根本问题（persona 取用户「第一性原理」提示词），后续枚举按根本问题展开——扫描质量不被主题措辞水平绑架；
1. **多模型合议**：同一枚举提示词并行发 DeepSeek + OpenAI + Gemini（可用几家发几家）；最快返回的先上墙，其余增量补充；
2. **三类卡强制配额**：每家产出必须覆盖 `学科视角 / 理论框架 / 研究方法` 三类；方法卡必须附一句可行性（数据从哪来：裁判文书、问卷、语料库、比较法样本……）；
3. 合并去重（规范化名称匹配）；**离群标亮**：仅一家模型提出的角度单独标记；
4. **新颖性三角定位**（每卡异步补挂）：定向检索「谁已经用 X 理论/方法研究过 Y」——
   - 英文面：Perplexity/Tavily；
   - **中文/自有面：`zsearch`**（本地 Zotero 语义检索，subprocess 调用，秒级零成本）；
   - 按命中密度分类：`主流（弃）/ 边缘有人做（可做深）/ 交叉空白（金矿）`；
   - **中英密度差信号**：英文成熟 × 中文法学冷 ＝ 经典引入型创新机会，单独金标。

**卡片 schema**：类型（视角/理论/方法）、理论或方法名、一句话机制、为什么对**你**（相对画像）非显而易见、**最强反驳**（steelman + 哪类审稿人会这么打）、文献锚点（真实检索）、新颖性分类、中英密度对比、离群/共识标记、来源模型、**三键反馈**（已知 / 新但不适用 / 新且值得深挖）。

**落盘**：`perspectives.md`（全部卡片，含反馈状态）、`questions.md`（每卡附 1-2 个拷问句作 grill-with-docs 弹药——**不开圆桌也有种子**）、`sources.md`（锚点）、`angle-feedback.json`（抑制表）。

## 5. 构思幕 · 圆桌深挖层（沿用 Co-STORM，扩展）

- 从卡片进入：以选中切入点为种子话题起 Co-STORM（热身 1-3 分钟可接受，首屏已有产出）。
- **钉死席位**：① 第一性原理专家（同 §4 步骤 0 的 persona）；② 跨学科猎人（只负责把外域理论拖上桌并论证适配性）。自动发现的领域专家照旧。
- 检索接地：多源检索层（Tavily / Perplexity / Mixed / Jina 全文增强，已在 main）。
- 主持人机制保留（信息层盲区挖掘）。
- 产物追加进 `questions.md` / `mindmap.md`。

## 6. 对抗幕（新建，轻量引擎，不套 Co-STORM）

- **有稿模式**：扫 `$PAPER_MUSE_OUTPUT_DIR` 下 .md（含 `01_成品稿/`）选草稿 → 抽中心主张 → 每条 3-5 个最可能崩的失败点 → 检索证伪/佐证 → 无证据标"未决"不放行 → `failure-points.md`。
- **无稿模式**：输入一个主线句（或从卡片一键送入）→ 同一红队流程攻击该主张——把反对观点提前到**动笔前**这个最便宜的干预点。
- persona 纲领取用户「对抗式审查」提示词原文（苛刻审稿人、3-5 失败点、拒绝无证据的"看起来没问题"）。
- 检索证伪同样走双面（英文 web + zsearch 中文/自有语料）。

## 7. 多服务商

- 横切 `provider ∈ {deepseek, openai, gemini}`：一键整体切换圆桌六角色（仓库已有适配）。
- 可选混编：高频角色用快模型，主持人/综合用强模型。
- **多模型合议独立于此开关**——扫描层始终尽可能多家并行；模型间分布差异是发现机制（离群角度标亮），不是成本选项。

## 8. 文件契约（联动的唯一接口）

`docs/agents/muse/`：`perspectives.md`、`questions.md`、`mindmap.md`、`failure-points.md`、`sources.md`、`profile.md`、`angle-feedback.json`。
消费者：`grill-with-docs`（吃 questions）、`to-prove`/`diagnose`（吃 failure-points）、paper-annotator（失败点锚回稿面）、任何后续 agent。联动靠文件约定，不靠共享界面。

## 9. UI / 审美路线

- SwiftUI 壳保留；内容画布 = WKWebView 加载本地 web UI（muse_server 静态托管）。
- **流式卡片墙是一等公民**：卡片逐张浮现、锚点/新颖性徽标异步补挂——架构上要求 server 提供增量接口（轮询增量或 SSE，实施计划定）。
- **设计流程**：先出 4 个截然不同设计方向的静态 HTML mock 供用户反应，再实现选中方向；本地预览-截图-迭代闭环。
- 视觉基调待 mock 阶段确认：现代、编辑器级排印、明暗双主题。

## 10. 与既有工作的衔接

- 多源检索层已合入 main（PerplexitySearchRM / JinaFullTextRM / MixedRM + server/REPL 接线，tests 6 项）。
- `zsearch` 为本期新增的本地依赖（用户已装，`/opt/homebrew/bin/zsearch`）；调用失败时中文面降级为"未知"并明示，不阻塞出卡。
- muse_server 新增接口（/scan 系、对抗幕、静态 UI 托管）由实施计划定义；实施计划按子系统拆分（见计划文档）。
- anamra 联动照旧（PAPER_MUSE_OUTPUT_DIR）。

## 11. 非目标 / 后续单列

- 不生成论文正稿文字（文字"活人感"归 humanize）。
- 不做云端/多用户/移动端。
- 本期不做：**CNKI 自动检索面**（需 opencli 浏览器会话，列后续计划——中文面先由 zsearch 承担）、Zotero VectorRM 全量混检入圆桌、思维导图可视化。

## 12. 验收标准

1. **质量地板（核心）**：前三次真实使用中，至少一张卡被用户标为「新且值得深挖」；
2. 速度：首批卡片 ≤20 秒上墙，全墙（含新颖性徽标）≤90 秒；
3. 每次扫描：三类卡齐备、≥1 离群标亮、每卡带最强反驳与真实文献锚点、中英密度对比字段非空（zsearch 不可用时明示"中文面未检"）；
4. 抑制表生效：标「已知」的角度在同论文再次扫描中不出现；
5. 对抗幕两种模式均产出 `failure-points.md`，每条失败点带证据链接或「未决」，无凭空断言；
6. `provider` 三选一均可跑通两幕；
7. `docs/agents/muse/` 七件产物齐备，grill-with-docs / to-prove 可直接消费；
8. App 视觉经用户按「4 方向反应法」定稿，用户认可"好看、现代"；全程无终端操作。
