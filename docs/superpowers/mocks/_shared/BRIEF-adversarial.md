# 共用简报：muse 对抗幕 mock（四方向反应法 · 第二季）

四个 HTML mock 共用这份简报 + 下方同一份写死数据。你负责其中**一个方向**，方向的交互骨架见
末尾「四个方向」。这份文件是**四个方向都必须满足**的公共契约。

（第一季 = 构思幕卡片墙 4 方向，见 `BRIEF.md`，用户已选定 **① 文稿台**。本季承其视觉语言。）

## 这是什么

**对抗幕** = 两幕剧的第二幕（第一幕构思幕 = 盲区扫描卡片墙，已实现）。红队审查：把「这里会崩」
提前到**动笔前**这个最便宜的干预点。轻量引擎（不套 Co-STORM）：抽中心主张 → 每条 3–5 个最可能
崩的失败点 → 双面检索证伪/佐证 → **无证据标「未决」不放行** → 落 `failure-points.md`。

本 mock 是静态视觉稿，给用户在浏览器里挑**交互/信息骨架**方向，**不接后端**（数据写死）。
最终归宿同构思幕：macOS 14+ 的 WKWebView 本地托管，**无外网依赖**。

## 与构思幕的关系（关键：这不是重选皮肤）

视觉语言**已锁定 = 文稿台**（朱批 / 单栏手稿 / 宋体正文 + 楷体嗓音 + 朱红批注）。四个方向：

- **共用 `_shared/tokens.css`**（内联抄入），**不得引入新色系 / 新字体 / 新顶栏**。
- 顶栏两幕切换 `[构思幕 | 对抗幕]` 保留，本季 **对抗幕 = 激活态**（构思幕变可点回的非激活 tab，占位即可）。
- **分歧点 = 红队审查「怎么摆上台面」的交互与信息骨架**，不是基础美学。四个方向是四种把「对抗」
  戏剧化的方式，共享同一套朱批语言。别把「换个配色」当方向差异。

## 两个模式（每个 mock 都要有，一个切换）

- **有稿模式**：扫 `$PAPER_MUSE_OUTPUT_DIR` 下 .md 草稿 → **草稿选择器**选一篇 → 抽中心主张（可多条，要能切/列）。
- **无稿模式**：**主张输入框**输入一句主线，或**从构思幕卡片一键送入** → 攻击这一条主张。
- 两模式切换后，共用下方「主张 → 失败点」的审查骨架。

## 每个 mock 都必须包含（缺一不可）

1. **顶栏**：两幕切换（对抗幕 active）+ 模式切换（有稿/无稿）+ 主题预填框 + 明暗切换按钮（沿用 tokens `.topbar`）。
2. **中心主张**显著呈现：可读的主张句，是「被告席」。有稿多条可切换/列表，无稿一条居中。
3. **每条主张 3–5 个失败点**，每个失败点一等公民，必须含：
   - **失败点陈述**（红队一句：哪里会崩）；
   - **失败类型**（样本偏差 / 内生性 / 概念滑坡 / 反例 / 机制缺环 / 规范·实证混淆 …）——标签化，一眼可分；
   - **严重度**（致命 / 重大 / 存疑）——要有视觉分级；
   - **裁决三态**：已证伪 / 有佐证 / **未决**。`未决` = 检索无证据，**必须最醒目、不可淡化**
     （本工具的灵魂：拒绝无证据的「看起来没问题」）。
4. **证据链**：非未决失败点挂 `evidence:[{title,url,stance:证伪|佐证}]`，真实文献锚点样式（可折叠但要在）；
   `未决` 显示「未检得证据 · 不放行」并暗示解锁（补检索/换词/需 CNKI 会话）。
5. **双面密度**：证据检索走双面（英文 web + 中文/自有语料），`en_hits`/`zh_hits` 成对呈现
   （`zh_hits=null` = 中文面未检），呼应构思幕的密度对比。
6. **persona 声音**：对抗式审查 = 苛刻审稿人的嗓音，用楷体/朱批承载「哪类审稿人会怎么打」，
   别写成中性 checklist。
7. **一键动作**：失败点 →「送 to-prove / diagnose」；无稿主张 →「从构思幕卡片送入」；
   产出 →「落 `failure-points.md`」的产物感（可复用右侧「附录·版记」抽屉语言，见 `ext-products-*`）。
8. **流式感**：红队逐条抛失败点、证据异步补挂（呼应扫描「先上墙后补徽标」），stagger 入场，
   尊重 `prefers-reduced-motion`。
9. **明暗双主题** + 可访问性地板（语义标签/焦点可见/对比度） + 响应式（宽内容各自 `overflow-x:auto`）。

## 硬约束

- **继承 `BRIEF.md` 全部硬约束**：单文件自包含（CSS 内联、数据内联、一个小 render 函数）、
  **零外链**（无 CDN/webfont/外图/fetch）、**仅 macOS 系统字体**、可访问性地板、响应式、
  **反 AI 设计俗套**——此处不复述，照那份执行。
- **内联 `tokens.css`**，只用其中变量与字族搭；顶栏沿用 tokens 的 `.topbar/.act-tab`。
  `未决` 用 `--vermilion` 家族但要与「已证伪/有佐证」拉开层级——**别三态一个红**（用实心/描边/淡染分级）。
- **反额外俗套（对抗题材易踩）**：别用骷髅/炸弹/红色警报灯 emoji、别整屏血红、别把「审查」做成
  杀毒软件的扫描进度条。对抗的重量靠**排印、留白、朱批的克制**，不靠恐吓色。

## 四个方向（每方向一个独立 HTML，一名 agent 认领一个；顶部注释写方向名 + 一句论点）

1. **对簿（法庭对峙）** — 主张 vs 失败点左右对峙：左「被告席」是主张，右是红队逐条「指控」，
   证据 = 呈堂「证物」，`未决` = 案上「悬而未决」。朱批 = 审判长旁批。大胆花在两栏对峙的张力。
   → `adv-01-docket.html`
2. **红笔审稿（稿面批注）** — 失败点作为**朱批锚回草稿正文的具体句子**（paper-annotator 的反向）：
   稿面为主体、失败点为浮标旁注，最贴「动笔前干预」，最大化复用文稿台的朱批旁注骨架。
   （无稿模式退化为单句主张 + 其下批注。）→ `adv-02-markup.html`
3. **拆解阵（主张解剖）** — 把主张拆成**承重结构**（前提 → 推理 → 结论），失败点按它攻击的结构位置挂载，
   红 = 高危未决的「应力点」。像结构工程的受力图，一眼看出主张哪根梁最先断。→ `adv-03-teardown.html`
4. **盘问流（逐条过堂）** — 一条竖直**审讯时间线**：红队逐条抛失败点、证据实时补挂（呼应扫描流式），
   你逐条批「认 / 驳 / 存疑」。单列、有节奏，一次只面对一条失败点。→ `adv-04-cross-exam.html`
